                                                                                                                                                          Lev Gorodinski Mod Nitin Babariya 5 years ago

                                                                                                                                                          Application services are at the edge of the domain. The implementation of UoW is an infrastructure concern, however the domain should define what it is, and the application service is a convenient place to do that.

                                                                                                                                                          To add some things I've learned since is that while dependency injection, as depicted here, is better than say service locator, an even better approach is to express the domain logic as a term language which you then provide interpreters for. So in PurchaseOrderService, rather than having the Invoice method orchestrate repositories and aggregates, you'd have a domain service which returns a value representing the result of an invoicing operation. In your application, you then provide an interpreter for this service which feeds inputs from wherever they need to come into the service, and then interprets the outputs by committing to a DB, etc. The challenge here is having a sufficiently expressive term language - if you just return a simple value, then its straightforward. If however you'd like to allow your term language to express requests for external services, yielding control to the interpreter and then resuming, you need more sophisticated mechanisms (such as free monads, free applicatives in an FP language). This approach truly isolates the domain language whereas DDD building block approaches as presented in this post are bound to make compromises.
